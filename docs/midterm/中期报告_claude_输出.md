### 页面标题
系统总体架构设计

### 架构图内容（需手绘或 draw.io）

```
Input → Master (调度+校验)
          │
          ├── Worker 1 (/infer 或 /infer_light) → Slice 1 (ONNX+EZKL)
          │     ↓ output_1
          ├── Worker 2 (/infer 或 /infer_light) → Slice 2 (ONNX+EZKL)
          │     ↓ output_2
          ├── ...
          └── Worker N (/infer 或 /infer_light) → Slice N (ONNX+EZKL)
                ↓ final_output

校验路径:
  L1: SHA-256(output) vs hash_out     — 每个节点
  L2: proof linking (Poseidon 哈希)    — 相邻 proof 节点
  L3: prev.hash_out == curr.hash_in   — 全链路
  Challenge: /re_prove                 — light 节点随机挑战
```

### 文字说明
- Master：统一调度推理请求，汇总结果，执行多层验证
- Worker：执行局部子模型推理，按角色生成证明或轻量响应
- proof 节点调用 `/infer`，生成完整零知识证明
- light 节点调用 `/infer_light`，仅做推理 + 哈希，可被随机挑战

### 本页结论
> 采用 Master-Worker 架构，将推理流与验证流分层解耦。

## 第 5 页：核心验证机制

### 模板版式
- 使用模板“分块说明”页
- 版面分成 L1 / L2 / L3 / 随机挑战 四块
- 本页不放已有实验图片

### 页面标题
分层可验证机制设计

### 层 1：外部完整性检查（L1）
- 对每个节点的输入/输出做 SHA-256 哈希校验
- 开销极低，可快速发现非恶意故障

### 层 2：零知识证明验证（L2）
- proof 节点在 Halo2 算术电路中生成 EZKL 证明
- Master 本地独立验证 proof 有效性
- proof-bound output：推理输出直接从 proof 公开实例提取，与证明数学绑定

### 层 3：跨节点哈希链（L3）
- 前一节点 `hash_out` 与后一节点 `hash_in` 对齐
- 检测链路传输异常

### 补充：随机挑战
- 对 light 节点发起 `/re_prove`，要求其重新生成证明
- 提供概率性威慑，补充 light 节点的安全约束

### 本页结论
> 分层验证：proof 节点提供密码学保证，light 节点提供轻量完整性检查。

| 机制 | 正常 proof 时间 (ms) | 相对 all_public 倍数 | 说明 |

# 口径建议

### 可以放心说的
- "系统已实现分布式推理原型，2/4/8 切片配置下均可正常运行"
- "选择性验证在保持边覆盖约束下，端到端开销降低约 36%~42%"
- "proof 节点上的响应层篡改被 proof-bound output 机制在源头预防"
- "hashed 模式约 1.6 倍时间开销，private 与 all_public 基本持平"
- "PyTorch 层级切片一致性验证误差为 0（bit-exact）"

### 必须加限定条件的
- 所有检测/预防结论前加："在当前攻击模型（响应层篡改）与实验设置下"
- 提到选择性验证时加："请求验证比例 ≠ 实际 proof 覆盖率，需以 actual_proof_fraction 为准"
- 涉及 L2 linking / 随机挑战时加："本次实验数据来自简化管线，完整校验效果需参考主系统"
- 提到分布式时加："当前为本地多进程模拟的研究原型，非多机实网部署"

### 绝对不要说的
- ❌ "完整全局密码学验证"
- ❌ "所有节点都被零知识证明严格约束"
- ❌ "100% 检测所有恶意行为"
- ❌ "25% 验证率就只验证了 25% 节点"（实际覆盖率更高）
- ❌ "系统已完全成熟 / 可直接部署"
